# 評価

通常のアプリケーションでは、`laravel-app`, `rails-app`のような、MPA+island(Vue)までが無難。

実装例を試せばわかると思うけど、SPA的な実装（SPA, Node SSR, Node SSG）は、どうあがいても正解が見えてこない。

MPA+island(Vue)は、最初から正解にいるという感じになる。

技術的に可能と実用的は別という感じになる。

ただし、+island(Vue)がないMPAも状況次第では不正解の場合もある。

## SPAの中での評価

SEOが不要な場合は、Vueだと、JSがクライアントのみになり構造がシンプルになるので、比較的管理しやすい。

SEOが必要な場合は、AstroかNuxtJSになる。

どちらもprefetchできるので、パフォーマンスは大差なさそう。

管理画面のURLをわかりにくくしたいときは、Astro、NuxtJSはファイルベースルーティングのため、実装例のようになる。

NestJSをAPIにすることで、すべてJSで実装できるので、フルスタックエンジニアを見つけやすいかもしれない。

## それぞれの評価

### SPA

#### メリット

- インタラクティブな実装が容易。

#### デメリット

- コスト
- SEO
- 全体を見通しにくくなる。

### Node SSR

#### メリット

- SEO
- インタラクティブな実装が容易。

#### デメリット

- [Node SSRの危険性](./nodessr.md)
- コスト

### Node SSG

#### メリット

- [Node SSRの危険性](./nodessr.md)がない

#### デメリット

- コスト
- SEO対策で、できることが限定的になる

### レガシーMPA

#### メリット

- コスト

#### デメリット

- インタラクティブな実装が、困難になる

### MPA+island(Vue)のようなハイブリッドMPA

#### メリット

- コスト
- インタラクティブな実装も可能。

#### デメリット

- インタラクティブな実装も含めると習得コストが高い。ただし、他の方法でも、全体を見る場合は同じなので、デメリットと呼べるかは微妙。


## コストの細分化

コストと言っても、いろいろある。

SPA的な実装では、複雑なためエンジニアの単価が高くなり、必要とする人材を見つけにくい。

さらに、障害発生率の上昇、原因の特定のしずらさ、複雑化するためのメンテナンスコストが上昇する。

Laravel, Railsだと、認証など基本的な仕組みが、システムごとで違いがなくなるため、新規参加者が慣れるまでの時間が短い。

SPA的な実装では、システムごとに異なってくるため、慣れるまでの時間がかかる。

慣れたころには、プロジェクトを離脱しているということにもなりかねない。

SPA的な実装での、コストの細分化をリストアップすると、こんな感じ。

- エンジニアの単価、見つけるコスト
- 新規参加者が慣れるまでのコスト
- 障害発生率
- 障害対応時間
- ランニングコスト

コストとできることのバランスが、ハイブリッドMPAが一番無難だと思う。
